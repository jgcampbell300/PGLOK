# PGLOK Release Checklist

## How to Create a Release

Creating a release is a **fully automated** process — push a version tag and let CI do the work:

```bash
# 1. Bump the version in src/__init__.py
vim src/__init__.py   # Change __version__

# 2. Update RELEASE_NOTES.md with changelog

# 3. Commit, tag, and push
git add -A
git commit -m "Bump version to vX.Y.Z"
git tag vX.Y.Z
git push origin main --tags
```

GitHub Actions will:
- Build Linux executable (`PGLOK-Linux-vX.Y.Z.tar.gz`)
- Build Windows executable (`PGLOK-Windows-vX.Y.Z.zip`)
- Package source archive (`PGLOK-vX.Y.Z-source.tar.gz`)
- Create the release and upload all assets

## Release Info

- **Tag format:** `vX.Y.Z` (e.g., `v0.2.4`)
- **Release title:** `PGLOK vX.Y.Z`
- **Target branch:** `main`
- **Notes source:** `RELEASE_NOTES.md`

## Release Assets (auto-generated by CI)

| Asset | Platform | Type |
|---|---|---|
| `PGLOK-Linux-vX.Y.Z.tar.gz` | Linux | .tar.gz (auto-update compatible) |
| `PGLOK-Windows-vX.Y.Z.zip` | Windows | .zip (auto-update compatible) |
| `PGLOK-vX.Y.Z-source.tar.gz` | Source | .tar.gz |

## Auto-Update Expectations

- Linux asset names must include `linux` and end in `.tar.gz`.
- Windows asset names must include `windows` and end in `.zip`.
- Source asset names must include `source`.

## Build Status

| Platform | Build Method | Status |
|---|---|---|
| Linux | Native PyInstaller on Ubuntu runner ✅ | Active |
| Windows | Native PyInstaller on Windows runner ✅ | Active (via CI) |
| macOS | Future — needs darwin runner | Planned |

## Local Windows Build

If you need to build the Windows .exe locally:

```bash
# On actual Windows:
build_windows.bat

# On Linux with Wine (limited — numpy native code may not work with Wine 9.0):
bash build_windows_via_wine.sh
```

## Verification

- [ ] Confirm the version displayed in the app matches the tag version.
- [ ] Launch the app and confirm normal startup.
- [ ] Test that auto-update offers the new version from a previous version.
